7. Deep link.
Related rules: `BR032`, `BR034`, and `BR035` are `confirmed`. `BR029` stays `supported_hypothesis`. `BR084` stays `not_testable`.
Files:
`app\weighttrack\resources\AndroidManifest.xml`

`app\weighttrack\sources\com\ikdong\weight\activity\MeasureActivity.java`
`app\weighttrack\sources\com\ikdong\weight\activity\NavigatorActivity.java`
Chain:
The Manifest exposes multiple `wta://` deep links, including callback routes for Fitbit and Withings and a measurement route. In the OAuth callback paths, the URI `code` value is read directly and passed into token-exchange logic. In the measurement path, `MeasureActivity` reads `PARAM_CATE` from the incoming URI and uses it to query and display local measurement data. The original mentor review kept a very specific wording here: the confirmed issue is that the sensitive measurement page does not go through the same unified pattern-lock or auth check as the `NavigatorActivity` path. That is a code fact. It is not proof that an outside attacker already dumped the full local database.
Limits:
There is no browser or adb reproduction in the evidence set, so the dynamic deep-link testing rule stays out.

9. Boundaries.
Main points:
BLE rules stay `not_supported` because the app does not show Bluetooth permissions or first-party BLE APIs.
Custom TLS rules stay out because no trust-all `TrustManager` or `HostnameVerifier` path was found.
ContentProvider rules stay out because the providers are not exported in the way those rules require.
The independent OAuth `state` issue, wide provider-path issue, and import-integrity concern remain separate engineering items and should not be rewritten as stronger confirmed rule conclusions.
