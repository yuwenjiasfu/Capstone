1. I checked the basic information of this app and found that the audit object is `cn.xiaofengkj.fitpro`, and the Manifest shows `versionCode=200`, `versionName=2.9.75`, `targetSdkVersion=35`. I think it is a BLE wearable/health management App with functions covering heart rate, blood pressure, blood oxygen, sleep, steps, body temperature, ECG, exercise track, weather, QR code payment, voice chat, OTA For advertising monetization, the first-party namespace is mainly `xfkj.fitpro.*`, the corresponding package name and provider authority is `cn.xiaofengkj.fitpro`, and the main backend is `https://fpapi2.jusonsmart.com`.

2. The second independent discovery is I-02. I noticed that WebSocket uses plain text `ws://` and puts the user ID and token into the URI path. I think this is not only a transmission encryption problem. Putting the bearer token derived value into the URI path will also enter the client log, proxy log, server access log, crash diagnosis and intermediate device record. Even if it is subsequently changed to `wss://`, the token still needs to be moved from the URI to the Header or the first encryption handshake message; the evidence is in `sources/xfkj/fitpro/websocket/JWebSocketClientService.java`, the specific structure is: ``` URI.create(String.format("ws://hiapi.jusonsmart.com:7780/websocket/%1$s/%2$s", Long.valueOf(DBHelper.getUserId()), DBHelper.getUserToken().replaceAll("-", ""))) ``` `onMessage()` in the same file will also directly write `Log.e("JWebSocketClientService", "message:" + str)` to write the server message body to the Android log.

The third independent discovery is I-03, I found that the fixed public Bearer Token is compiled into the client, `Constant.TOKEN` is a static string `"Bearer 6fcb7f58475b4e5aad8f0f1cadce235e"`, `DBHelper.getUserToken()` will fall back to this when there is no user session token, `HttpHelper.updateDeviceInfo()` also directly uses this token to upload device information. I think if the server uses it as a trusted client credential, anyone who obtains the APK can extract and reuse it; the evidence is in `sources/xfkj/fitpro/utils/Constant.java`, `sources/xfkj/fitpro/db/DBHelper.java`, `sources/xfkj/fitpro/api/HttpHelper.java`, `updateDeviceInfo()` Use `Constant.TOKEN` to upload MAC, device model, system version, language, mobile phone model, user ID. Even if the token is just an "anonymous app token", it still requires server-side permissions, rotation, source verification and abuse detection.

The fifth independent discovery is I-05, I noticed that there is a large integration of advertising and statistics SDK in the health app. Manifest and resource display integration Google Ads/Firebase, AppLovin, InMobi, Yandex, Unity Ads, Facebook Audience Network, Vungle, IronSource/LevelPlay, ByteDance/Pangle, Mintegral/MBridge, Bigo, Amazon APS, ThinkUp, smartdigimkt, Aliyun SLS, etc.; I feel that a single SDK name cannot prove leakage, but in health/wearable applications this will significantly increase the third-party processing surface. Without dynamic traffic, this should only be written as a cautious risk, not as a confirmed leak.

5. The sixth independent discovery is I-06. I found that the local database and logs contain health and identity data, and the DB is clearly not encrypted: `DBModule.ENCRYPTED=false`. The GreenDAO table contains heart rate, blood pressure, blood oxygen, sleep, body temperature, ECG, exercise, GPS, token, personal information, device MAC, etc. The database directory is used first. `Environment.getExternalStorageDirectory()/Android/data/<package>/db`; I think modern Android has access restrictions to `Android/data`, but it is still not an internal encrypted storage protected by SQLCipher or Keystore. Combined with `requestLegacyExternalStorage`, external read and write permissions, and log export functions, the risk is higher than ordinary internal private storage.
