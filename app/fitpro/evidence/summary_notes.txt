Fitpro.

1. The cleartext transport issue is real here.
Rules: BR006 and BR013.
Status: confirmed.
This one should stay confirmed. It is not only a weak manifest flag. It is also used by first-party code. `app\fitpro\resources\AndroidManifest.xml:154` sets `android:usesCleartextTraffic="true"`. `app\fitpro\resources\res\xml\network_security_config.xml:3` and `:13` both allow cleartext traffic. `app\fitpro\sources\xfkj\fitpro\websocket\JWebSocketClientService.java:42` builds `ws://hiapi.jusonsmart.com:7780/websocket/%1$s/%2$s`. The first path value comes from `DBHelper.getUserId()`. The second path value comes from `DBHelper.getUserToken().replaceAll(...)`. So the account identifier and token-derived value are part of a plaintext WebSocket design. That is enough to keep BR006 and BR013 confirmed. I am not stretching this into a claim that every business API uses plaintext. I am also not stretching this into a claim that HTTPS itself is broken.

2. BR095 still should not be upgraded.
Rule: BR095.
Status: highly suspected only.
The original review was careful on this point and I am keeping that. The app clearly allows cleartext and clearly uses a plaintext WebSocket. But I did not see a first-party permissive TrustManager. I did not see a first-party HostnameVerifier that always returns true. I did not see a first-party HTTPS downgrade path from TLS failure to HTTP. So the stronger "weak HTTPS validation" style claim still does not cross the line. The confirmed part is the plaintext WebSocket and the global cleartext-allowed posture. The rest stays below confirmed.

5. The BLE command abuse angle still stays below confirmed.

6. Local storage protection is weak in both the database and preference layers.
Rules: BR025 and BR026.
Status: confirmed.
`app\fitpro\sources\xfkj\fitpro\db\DBModule.java:32` sets `ENCRYPTED = false`. `app\fitpro\sources\xfkj\fitpro\utils\PathUtils.java:30` defines the root under `Android/data/<package>`. The same file at `:57` builds the DB directory. The same file at `:101-102` points to external storage plus that app directory. The original report already mapped the schema to health, identity, token, password, device, and location fields. That is enough to keep BR025 confirmed. The preference side is also not protected. The original notes already showed ordinary SharedPreferences storing Bluetooth address, body values, goals, H5 URL, and other state. So BR026 also stays confirmed. I am not exaggerating this into "any app can read everything on every modern Android version". The point is weaker storage design, not universal cross-app access on all devices.

7. The external app-specific storage point should stay confirmed, but with the same careful wording as before.
Rule: BR027.
Status: confirmed.
The files are not described as being in a fully public root folder. They are described as being in external app-specific paths. That distinction matters and I am keeping it. `app\fitpro\sources\xfkj\fitpro\utils\PathUtils.java:30` and `:101-102` show the external app-specific base. The DB sits under that area. The WebSocket voice AMR files also go to an external app music path. The original review already said not to overstate this as universal public storage exposure. I agree with that limit. But the combination of unencrypted DB, exported functions, log files, and app-specific external placement is still enough to keep BR027 confirmed.

8. Backup protection is also weak.
Rule: BR028.
Status: confirmed.
`app\fitpro\resources\AndroidManifest.xml` enables backup according to the original audit. `app\fitpro\resources\res\xml\backup_rules.xml:2` is just `<full-backup-content/>`. `app\fitpro\resources\res\xml\data_extraction_rules.xml:3` allows `<cloud-backup/>`. The key point is that I did not see exclusions for sensitive DB or session material. So the backup exposure finding still stays confirmed. I am not claiming that cloud backup definitely happened on a real device. I am confirming that the shipped backup policy does not carve out the sensitive material.
