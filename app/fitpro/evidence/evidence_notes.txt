1. The cleartext WebSocket account chain.
Related rules: BR006 and BR013 confirmed. BR095 highly suspected only.
Files:
`app\fitpro\resources\AndroidManifest.xml:154`
`app\fitpro\resources\res\xml\network_security_config.xml:3`
`app\fitpro\resources\res\xml\network_security_config.xml:13`
`app\fitpro\sources\xfkj\fitpro\websocket\JWebSocketClientService.java:42`
`app\fitpro\sources\xfkj\fitpro\websocket\JWebSocketClientService.java:45`
`app\fitpro\sources\xfkj\fitpro\websocket\JWebSocketClientService.java:69-75`
Package:
The application globally allows cleartext. The network config also allows cleartext at both the domain and base levels. The first-party WebSocket service does not use `wss://`.
Code:
It builds a URI with `ws://hiapi.jusonsmart.com:7780/websocket/%1$s/%2$s`. The first value is `DBHelper.getUserId()`. The second value is `DBHelper.getUserToken().replaceAll(...)`. So the path itself carries account-linked material. `onMessage(String str)` logs the server message. The service later decodes audio content and writes an `.amr` file.
Basis:
This is not just a policy weakness. It is an actual first-party plaintext design. The token-derived value is in the URI path, not just buried inside encrypted payload later.
Limit:
I still do not have runtime packet capture. I still do not claim every health API is plaintext. I still do not claim HTTPS validation is broken everywhere.
