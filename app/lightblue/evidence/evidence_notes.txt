3. BLE value to Adafruit bridge entry.
Related rules: BR091, BR097, and BR098.
Status: highly suspected.
Files:
`app\lightblue\sources\O5\a.java:72`
`app\lightblue\sources\P5\a.java:56-66`
`app\lightblue\sources\P5\b.java:28`
`app\lightblue\sources\P5\f.java:235`
`app\lightblue\sources\com\punchthrough\lightblueexplorer\ui\characteristicdetail\b.java:779-808`
`app\lightblue\sources\com\punchthrough\lightblueexplorer\ui\characteristicdetail\b.java:964-972`
`app\lightblue\sources\com\punchthrough\lightblueexplorer\ui\characteristicdetail\b.java:1324-1329`
Step by step:
The app sets the Retrofit base URL to Adafruit IO. The Retrofit interface defines the POST path for `{username}/feeds/{feedName}/data`. The Adafruit secret is sent in `x-aio-key`. The payload object is a single `value`. The characteristic detail page receives incoming bytes and stores them in the current view state. It checks whether the current characteristic UUID is already mapped for Adafruit. If there is a mapping, it calls `n0(byte[], UUID)`. `n0(...)` pulls the feed mapping and starts the upload coroutine. `l0(...)` is the path that binds the current characteristic to username, secret, type, and feed.
Impact:
This is not just a general cloud SDK in the package. It is a first-party UI and BLE path that can forward characteristic values to a third-party cloud.
Limit:
I did not observe a live POST request. I did not confirm a real feed configuration on disk. I did not confirm that the values in a given session were health data rather than generic BLE data.
