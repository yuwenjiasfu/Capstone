14. I think the problem with MQTT is that the MQTT example Activity uses `tcp://`, and the resource defaults to `tcp://iot.eclipse.org:1883`. This is plain text MQTT without TLS. Although it cannot be confirmed that it is a production entry, it remains in one namespace and reads `mqtt_usr` and `mqtt_psw`. The risk should not be ignored; it is recommended that if the MQTT example does not belong to production functions, remove it Activity, layout, string, Paho service dependencies; if it is a production function, change to `ssl://` or `wss://`, verify the certificate, disable the default public broker, and use Android Keystore/EncryptedSharedPreferences for credentials.

16. I think in terms of Wi-Fi sockets, the ordinary TCP service with fixed port 5373 does not see TLS, HMAC, challenge response or access control with the client. 17. I think the problem with the ProtoFeed protocol is that it uses `STX/ETX` frames and JSON. HASH is keyless MD5. Keyless MD5 can only detect accidental corruption and cannot prevent attackers from recalculating HASH. The message code generated by `Random().nextInt()` is not an authenticated random number. The suggestion is to introduce a device-level key or a shared application layer key after pairing, and use HMAC-SHA256 for each message, including nonce, timestamp, direction, command type and payload. Confirm business permissions and user for control commands. Maintain replay protection on the server/device side. The symmetric key should not be hard-coded in the APK. It should be derived through the registration/binding process or issued by the server and saved in Keystore.

18. I think `LocationUpdatesService` is exported but does not have component-level permissions on the component exposure surface. If you look at the external startup statically, it will enter `onCreate()` and be foregrounded. Continuous positioning is usually called `requestLocationUpdates()` after binding. This is not a confirmed location leak, but it expands the reachable surface. It is recommended to set `android:exported="false"` unless there is a need for cross-App use. If you must export, signature permission is required. Verify the calling source and action in `onStartCommand`, and stop the unknown action directly.
