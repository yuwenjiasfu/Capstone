I found that the local web service `XdripWebService` has an open mode combination risk, it is a first party local HTTP/HTTPS service, whether it is enabled by default depends on the preference, so I can't directly confirm the remote exposure, but the code shows that once the user enables "open" mode and does not set the webservice secret, the remote request can access the route without `api-secret`, and the response has `Access-Control-Allow-Origin: *`, which I feel is an obvious configuration combination risk.

4. I found that setting the export function to copy the internal `shared_prefs` XML to the public Downloads directory has the problem of leaking plain text sensitive fields. I think this is not a single `WRITE_EXTERNAL_STORAGE` permission issue, but the first-party code explicitly copies the internal `shared_prefs/<package>_preferences.xml` to `Environment.getExternalStoragePublicDirectory(Environment.DIRECTORY_DOWNLOADS) + "/xDrip-export"`, and there is no encryption and no field filtering.

7. I found that `ExternalStatusBroadcastReceiver` is a first-party export receiver that accepts external status lines by default and writes Intent extras to persistent storage. It also parses TBR/absolute basal to create APStatus records. I think this is not necessarily a vulnerability because it may be designed for compatible applications, but from an attack surface perspective it is unauthorized external input entering the health/treatment status field.

8. I found that the application uses Microsoft AppCenter Analytics/Crashes and `enable_crashlytics` is started in `Application.onCreate` by default, specifically `xdrip\sources\com\eveningoutpost\dexdrip\xdrip.java:69-70` is called when `enable_crashlytics` is true `AppCenterCrashReporting.start(this)`, `xdrip\sources\com\eveningoutpost\dexdrip\utils\AppCenterCrashReporting.java:47` Start `Analytics.class` and `Crashes.class` with fixed AppCenter app secret, `resources\res\values\strings.xml:174`, `:1246`, `:1255` There are related documents about automatic crash reporting and sending logs to developers. I feel that without runtime traffic, it is impossible to confirm that health data has been uploaded. However, enabling third-party crash/analysis SDKs by default is a privacy review item that should be followed up. Crash stacks, device information, and event logs may include device models, times, and error contexts. If the request body or health data is mixed in the business log, secondary leakage will occur. The repair recommendation is to change to explicit opt-in, or at least not initialize Analytics before the first startup, and perform a unified sensitive information scrubber before crash reporting.

9. I found that Tidepool OAuth AuthState is persisted in ordinary SharedPreferences. The new process of Tidepool OAuth uses AppAuth and PKCE, which has positive protection against deep chain attacks. However, the possibility of AuthState JSON containing access/refresh token is high. It is currently stored in default SharedPreferences through `Pref.setString`. If device root, backup/export, debugging or other local attacks are established, the token may be replayed.
