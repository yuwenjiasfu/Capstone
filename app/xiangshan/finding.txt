1. I found that the high-frequency namespaces of third-party SDKs/libraries include `cn.jpush`, `cn.jiguang`, `com.umeng`, `com.kwad`, `io.dcloud`, `com.tencent.bugly`, `com.huawei.*`, `com.google.*`. I think these SDKs are the main focus of subsequent permission and identifier collection risks.

2. I found that important domain names/URLs include business API `smartlink.senssun.com`, `global.senssun.com`, remote H5 `sharecn.senssun.com`, AI service `aihelper.senssun.com`, speech recognition `nls-gateway-cn-shanghai.aliyuncs.com`, and the HTTP business domain name of dev/test also exists in the assets, which is consistent with `usesCleartextTraffic=true` together form the risk context of the transport surface.

3. I found that what was replaced here was not the cached image, but the UNI Web front-end package that the App actually runs. There is a serious problem: once the local ZIP is replaced, the new H5/JS logic will obtain the original login status, request header, BLE JS Bridge, and offline data access capabilities in the App process. I think the attack surface has changed from "external files are read" to "external files become the execution content of the App." The chain of evidence is: App starts and enters `MainActivity.uniHome()` -> The first-party code actively checks `app.zip` in the external path -> If the file exists, first delete `www/` in the current running directory -> Then decompress the external ZIP to `www/` -> Subsequent `CustomUniActivity`/`PandoraEntryActivity` loads the replaced web resource -> The new resource can be directly reused in the App token, local cache, BLE JS callback capabilities and remote API capabilities that already exist in it.

4. The second independent observation I found is that the remote points H5 page has obtained the App login token and accountId, and there is a strong cross-layer trust boundary expansion problem. In `app-service.js`, `useBaseStore` saves the login credentials in the local storage key "Login Credentials", the account store saves the account information in the local storage key "Account Information", `pages/nvue/member-points/web.js` spells `token`, `accountId`, `statusBarHeight`, `version`, `windowHeight` into the URL when assembling the remote page address. The production environment address is `https://sharecn.senssun.com/smartlink-points/index.html?...#/pages/index/index?...`.

5. I found that what is passed here is not just the UI parameters, but the authentication token and account ID. Even if the token falls after the URL hash/fragment, it will not necessarily be sent to the server as the query of the initial HTTP request. It will still be directly read by the remote H5 page's own JavaScript. I feel that the risk surface has expanded from "native App code" to "remote H5, its dependent scripts, its CDN, and its publishing system." The evidence chain is: `useBaseStore` reads login credentials from local storage -> account store holds `accountInfo.accountId` -> `member-points/web.js` constructs `o={token, accountId, statusBarHeight, version, windowHeight, page}` -> code serializes `o` and appends it to the remote `sharecn.senssun.com` page URL -> the remote H5 page is controlled by the server, and the page script can directly read these parameters -> remote H5 can then call the App backend interface or perform account state operations based on the token. Common cloud-upload or deep-link categories do not describe this exact design very well, so I keep it as an independent concern.

6. A third independent observation I made was that the WebView certificate error allowed the user to continue accessing, which I felt was an obvious TLS weak design. The first-party evidence is in `sources/com/xs/smartlink/ui/WebActivity.java:77-85`. When the WebView certificate error is triggered, the code is allowed to go `sslErrorHandler.proceed()` after the pop-up box, and the WebView is turned on. `setAllowFileAccess(true)`, `setJavaScriptEnabled(true)`, `setGeolocationEnabled(true)`, `setSavePassword(true)`, `setMixedContentMode(0)`.

7. A fourth independent observation I made is this: the request signature scheme is a fixed MD5 variant visible to the client and should not be considered strong authentication material. The signature function in `app-service.js` concatenates the string with fixed constants `senssunhealth` and `senssuntwice` to do MD5. `getHeader$2()` uses `token`, `time` and `sign` as unified request headers. The same style `sign=a$2(token+time)` is also used for image upload.

8. I found that the App evidence of evidence chain B is as follows: local library name `smartlink_v1`; user table `user_v2` field `userId/accountId/name/birthday/height/weight/bmi/activity/userType/memoryValue`, upload interface `/data/v1/offline/user/data`; device table `device_v1` field `accountId/deviceId/productId/supplierId/serialNumber/unit/keyWords/source/createTime`, upload interface `/data/v1/offline/device/data`; body fat record `bodyFatScaleRecords_v5` field `bodyFat/bmi/bmr/bodyAge/boneMass/deviceId/heartRate/...`, upload interface `/data/v1/offline/four/historyData`; blood pressure record `bloodPressure_v1` field `historyId/userId/heightPressure/lowPressure/recordTime`, upload interface `/data/v1/offline/bloodPressure/history/data`; heart rate record `heartRate_v1` field `historyId/userId/heartRate/measureInfo/measureQuality/recordTime`, upload interface `/data/v1/offline/heartRate/history/data`; the request layer uniformly appends `token/time/sign` and uses `uni.request` uniformly. I think this chain connects "local privacy data risk" and "cloud upload risk". As long as you export the local database from the real machine and cooperate with packet capture, static analysis can be upgraded to dynamic confirmation very quickly.
