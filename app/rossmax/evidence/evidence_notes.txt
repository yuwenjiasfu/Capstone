Chain:
The health chain begins in `DeviceScanFragment`, where the app installs a BLE scan callback, builds Rossmax device filters for pulse oximeters, blood pressure devices, thermometers, scales, and blood glucose devices, and then starts scanning. When the user selects a device, the code saves the observed device and keeps its MAC address as measurement context. `VUBleManager` then exposes typed callbacks for blood pressure, glucose, pulse oximetry, temperature, and weight data. The delegates call `MeasureAdapter` conversion helpers, and those helpers turn the raw measurement objects into the app's own `Vital` models.
The model side is concrete. `Vital` carries `type`, timestamps, `measureTime`, `deviceSKU`, `deviceRecKey`, `remark`, and remove state. `BpData`, `BpExtension`, `GlucoseData`, `Spo2Data`, and `FireWeight` add the measurement-specific values. From there `DataManager.insertVitalData(...)` hands the object to `FirebaseVitalStore`, which writes into `users/{uid}/measurements/`, updates or deletes those records, and also keeps `spo2Detail` subcollections. That is why health-data handling itself stays confirmed, while cloud upload remains `supported_hypothesis` because no runtime network evidence exists.
Limits:
There is still no packet capture, no Firestore console output, and no runtime payload sample. The code proves the sink path exists, not that a specific real user record was observed in transit.

The Manifest registers `SplashActivity` for `https://r.rossmax.care` and enables `autoVerify`, but it does not restrict the path. `QRCodeFragment` and `FirestoreUtil.createInvitation(...)` build invitation objects, and `DeepLinkUtil.generateDeepLink(...)` turns the invitation id into a URL under `BuildConfig.DEEP_LINK_URL`. That is why the original report kept the deep link finding below confirmed. The path is real, but the guards and confirmation steps mean the present evidence does not prove unauthorized execution.
Limits:
The asset link verification state was not checked online, so `R027` remains out. There is also no runtime deep link reproduction showing a bad path or a bypassed confirmation.

7. BLE limits.
Main points:
The most important limitation is still the missing runtime BLE evidence. There is no sniffer capture, no real peripheral test, no same-device malicious app test, and no replay or injection proof. The negative static evidence does matter: the visible measurement path does not show app-layer MACs, signatures, per-measurement attestation, or another clear provenance check before data becomes a `Vital` and reaches Firestore. But the report should still keep those BLE rules at `supported_hypothesis`, not confirmed.
The original database-outside notes also matter here. Resource keys and the `vuioht` native decrypt path were treated as SDK configuration or license handling, not as proof that BLE measurement payloads use strong application-layer protection.
