1. I also saw log leaks in the BLE callback: Line 197 of `UHBleCallbacksHandler.java` records the BLE address, Line 274 records the blood glucose data JSON, Line 299 records the body temperature value, Lines 44, 56, and 232 of `MoxManager.java` records receiving/sending bytes and parsing data; I think `accessToken` is the account access credential, `uploadDataArray` Local tables such as blood pressure, blood sugar, blood oxygen, and member information belong to health and identity-related data. BLE MAC/device name can be used for device binding, tracking, and replay scenario analysis. The problem is that the production environment should not record tokens, complete health measurement JSON, and BLE MAC. It is recommended to desensitize and minimize logs before uploading. The server should also reject log packages containing tokens. The log directory and retention period should be included in the privacy policy and user-controllable deletion process.

2. I checked the Manifest's permission list. BLE/location-related ones include `BLUETOOTH`, `BLUETOOTH_ADMIN`, `BLUETOOTH_CONNECT`, `BLUETOOTH_SCAN`, `ACCESS_COARSE_LOCATION`, `ACCESS_FINE_LOCATION`, storage-related ones include `WRITE_EXTERNAL_STORAGE`, `READ_EXTERNAL_STORAGE`, and network-related ones. `INTERNET`, `ACCESS_NETWORK_STATE`, `ACCESS_WIFI_STATE`, `CHANGE_WIFI_STATE`, I think the high-risk permissions that need additional explanation are `READ_PHONE_STATE`, `SYSTEM_ALERT_WINDOW`, `READ_CALENDAR`, `WRITE_CALENDAR`, `CALL_PHONE`, `RECORD_AUDIO`, `WRITE_SETTINGS`, among which the BLE health device App itself requires some permissions, but READ_PHONE_STATE, SYSTEM_ALERT_WINDOW, WRITE_SETTINGS It still needs to be further verified during runtime and product description.

3. I sorted out the first-party API: `SyncAPI.syncData` is `POST api/datasync/index` with Header `Authorization`, Body is `SyncDataRequest`; `AccountAPI.login` is `POST api/account/login`; `AccountAPI.refreshToken` is `POST api/account/refreshtoken`; `AccountAPI.uploadPhoto` is `POST api/user/uploadphoto`; `UploadAPI.uploadLog` is `POST api/logger/upload`; `HoDeviceAPI` handles device binding, operating data, mode control, switch/oxygen concentration/negative ion/timing/voice and other interfaces, based on `HOST_HO`; `TerminalAPI.updateMac` is `POST api/terminalbindingble/updatemac` is used for BLE MAC update; the network client is `ServiceGenerator` using OkHttp/Retrofit and `TokenAuthenticator`. I did not find clear evidence of permissive TrustManager and HostnameVerifier returning true or downgrading HTTP after SSL exception in the first-party network client. Business URLs are mostly HTTPS; `http://www.yuwell.com` appears in SOAPAction. namespace but calling the endpoint is still spliced with `GlobalContext.getHost()`. I think the SOAP namespace cannot be regarded as an HTTP business endpoint.

When I combed through the local storage, I found that SharedPreferences `PreferenceSource` uses `SharedPreferencesUtil` to save `access_token`, `refress_token` (the original spelling is this), `encrypt_token`, `current_familymember`, and also save BLE/device status `bg_device`, `bo_device`, `bp_device`, `mox_device`, `fht_device`, `last_obj_*`, `last_*`, etc., I don't see Jetpack Security, EncryptedSharedPreferences or custom file encryption wrappers; SQLite side `DatabaseServiceImpl.getDbName()` returns `uhealth.db`, `DatabaseManager` uses xUtils `DbUtils.DaoConfig` creates a normal database and does not see SQLCipher/key configuration.

5. I further looked at the sensitive tables and fields and found that `BP_MEASUREMENT` contains SBP, DBP, pulseRate, deviceSN, terminalMAC, memberId, `BG_MEASUREMENT` contains measureValue, medicine, measureTime, deviceSN, terminalMAC, memberId, `OXY_DATA` contains spo2, heartRate, deviceSN, terminalMAC, memberId, `FAMILY_MEMBER` contains birthday, height, weight, waistline, phone, photo, nickname; in terms of logs, `Logger` configures both Android log and external-cache file logs. `SyncService` records decrypted download data, plaintext JSON before uploading, and map containing accessToken. `LogUploadUtil` compresses the log directory and uploads it to `api/logger/upload`. `CrashLog` writes to the `uhealth` directory under external storage when external storage is available.

6. I sorted out the BLE evidence: in terms of scanning, `BleScannerPresenter` and `UHBleMulticonnectService` use Android BLE scanning. Device identification mainly relies on device name, MAC, manufacturer data and type prefix. `UHBleMulticonnectService.getDeviceType()` identifies devices such as blood sugar, blood pressure, body fat, blood oxygen, body temperature, fetal heart rate, etc. by name; in terms of receiving and warehousing, blood pressure passes `onBloodPressureMeasurementRead` is transferred to `BPMeasurement`, blood sugar is transferred to `BGMeasurement` through `onGlucoseMeasurementRead`, blood oxygen is transferred to `saveOXYData` through `onOxygenDataRead` to generate `OxyData`, body temperature/fetal heart rate also has corresponding EventBus distribution; control aspect `MoxManager` is exposed Write commands such as powerOff, pullData, reset, setCustomModel, setLock, turnLight, etc. Custom UUID and characteristic values are visible in the code. Currently, there is no application layer HMAC, signature, device-side challenge-response or session token.

10. I sorted out the Mox control link: `MoxManager` lines 87-90 obtain the custom characteristic, line 79 enables notifications, lines 185-221 expose powerOff, pullData, reset, setCustomModel, setLock, turnLight, lines 231-233 only record bytes before writing the characteristic and then call the parent class to write, no app-specific challenge, HMAC, signature or session token; the conservative conclusion is that the current code is sufficient to support the "risk hypothesis of missing BLE application layer authentication" and cannot confirm that "any device can forge measurement values" or "an external attacker can remotely control the device" because it requires BLE devices, protocol packet capture and attack experiments.
