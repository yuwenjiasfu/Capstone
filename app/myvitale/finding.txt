I think the evidence chain of IF-01 is: after the user logs in successfully, `AuthPresenter.java:160-164` takes out the token from `LoginResponse` and calls `sessionManager.saveToken(it.getToken())`; `SessionManagerSP.java:32-34` writes the token into `session_local_store/token` as a normal string; at the same time Manifest Allow backup; if device/system backup, debug backup, vendor cloud backup, or forensic scenarios have access to the App data directory, tokens and configuration values may be restored. The current sample does not have a real device backup export, so it cannot be proven that it has been leaked, but the engineering combination itself is unreasonable. The token is authorization material, and `measuresServerUrl` determines the basic URL of authentication, user, data, and measurement upload API. Although the partner key is not proven to be a BLE protocol key, it is an SDK/business access credential. I recommend using encrypted storage supported by Android Keystore (such as EncryptedSharedPreferences) for tokens. `allowBackup` should be changed to `false` in the release App containing session credentials, or at least use `fullBackupContent` to exclude sensitive SharedPreferences, and perform device-side verification of data remaining after logout.

4. The third independent issue I found, IF-03, is that the preferences page allows arbitrary API Base URLs to be written, which is a medium risk. The authentication page has whitelist verification, but the user list page and preference page paths do not call `ApiUrlsValidator`, and the input URL will be saved directly. The whitelist validator location is `ApiUrlsValidator.java:20-29`, which only accepts URLs that are exactly equal in `R.array.clients_api_urls` (i.e. `https://vivagympre.myvitale.com`, `https://forus.myvitale.com`, `https://vivagym.myvitale.com`); the authentication page is at `AuthPresenter.java:197-205` has been verified, and it will be saved only if it is not empty and passes `apiUrlsValidator.isValid(url)`, otherwise it will prompt that it has not been verified; but the user list page `UsersListPresenter.java:392-399` only checks the empty string, if it is not empty, it means `preferenceManager.setMeasuresServerUrl(url)`; the preference page `PreferencesActivity.java:113-133` also only checks the empty string. If it is not empty, it will be written to `measuresServerUrl`.

I found that all API clients read `measuresServerUrl` as the base URL: `VitaleAuthService.java:242-263` is constructed through `URLParserKt.takeFrom(url, measuresServerUrl)`; `MeasuresApi.java:174-177` reads and sets path; `ProfilesApi.java:174-177` reads and sets path;`UsersApi.java:218-221` reads and sets path. So the evidence chain is: the user or automation enters a non-empty URL, `UsersListPresenter` or `PreferencesActivity` is saved directly to `BleConnectionPref/measuresServerUrl` without going through `ApiUrlsValidator`, subsequent authentication, user query, data update, measurement upload API all read the base URL from `PreferenceManager.getMeasuresServerUrl()`; if URL Directed to an attacker-controlled server, login, user profile, and measurement upload requests may be sent to unintended hosts. The limitation is that the current sample has no evidence of UI operation or packet capture. The authentication page path itself is whitelisted, and Android/Ktor will still perform HTTPS certificate verification by default. I suggest that the URL input portal reuses `ApiUrlsValidator` uniformly, and the release version is best not to expose the editable base URL.
