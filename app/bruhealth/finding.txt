1. The object of my review this time is the Android application bruhealth. The package name is `egnc.moh.bruhealth`, versionName is `5.1.8`, versionCode is `50108018`, and `targetSdkVersion=35`. It's actually a hybrid: native Android + Flutter + Cordova/WebView, with a bunch of vendor SDKs sandwiched between it. The functional aspects are very typical: steps, calories, heart rate, sleep, exercise distance of Health Connect/Google Fit; movement trajectory and positioning; blood glucose meter BLE; medical card, PHR, medication reminder, and H5/Flutter routing such as Warfarin. I will treat `egnc.moh.bruhealth` and `egnc.moh.base` as the first party, `com.evyd.*` as the supplier/quasi-first party (it does health, logs, Flutter, UI, and permission components), `com.bruhealth.glucometer.*` as the blood glucose meter business module, and not interfere with the advertising SDK. Third-party SDKs include Google/Firebase/ML Kit/Health Connect/Google Fit, Tencent TRTC, Cordova, inuker BLE library, marianhello background-location, OkHttp/Retrofit, Glide/Picasso, blankj utilcode, sqflite, etc.

2. I also saw a misspelled permission `android.android.permission.BLUETOOTH_SCAN` in the Manifest, which is I-04. `bruhealth/resources/AndroidManifest.xml` appears both `android.permission.BLUETOOTH_SCAN` and `android.android.permission.BLUETOOTH_SCAN`. Although misspelling will not grant system capabilities, it indicates that there are quality issues in the manifest permission configuration, which will interfere with permission audits, compliance instructions, and least privilege judgments.

3. I further looked at `bruhealth/sources/com/evyd/log/EvydLog.java`: `initLogUtilsForProd()` calls `LogUtils.getConfig().setLog2FileSwitch(true)`; `initLogUtilsForOthers()` calls `setLogSwitch(false)` and `setLog2FileSwitch(false)`. The combination of naming and call points is actually a bit counterintuitive - the one called ForProd opens file logs, but is called in a non-Prod environment. Then `bruhealth/sources/egnc/moh/base/net/ApiClient.java` uses `HttpLoggingInterceptor.Level.BODY`. When not debugging, `EvydLog.d2File("Http", strFilterFileContent)` is also called. Only the multipart file content is replaced. Ordinary JSON body, headers and URL will not be desensitized as a whole. The request headers of `bruhealth/sources/egnc/moh/base/net/HeaderInterceptor.java` include `wlapp-token`, `device-id`, `old-device-id`, `device-name`, `tid`.

4. I think the ID number `id_no`, mobile phone number `phone_no`, member ID `member_id`, and user ID `user_id` are identification data, which are associated with functional accounts in the health app and are more sensitive. Instead of just seeing SQLite dependencies, there is Room builder, table structure, field semantics, and DAO write paths; no encrypted database constructs are seen. The limitation is that I don't have the device database file and can't see the real content. However, the threat model must be written accurately: Manifest `android:allowBackup="false"`. Android private directories are not directly readable by other ordinary apps by default and cannot be written as "readable by any app". The accurate threat model should be root, debug backup, device compromise, log recovery link, and local forensics.

I string the token/cookie chain: after successful login, `Model.UserData.save(...)` saves the token to SharedPreferences; `AccountManager.getTokenString()` reads the token; `HeaderInterceptor` adds the token to each main API request; the cookie jar uses `PersistentCookieStore` to serialize the cookie to `CookiePrefsFile`. Instead of just seeing the `token` string, the saving, reading, request header usage and cookie store codes are all there. I did not extract the real SharedPreferences file, cannot enumerate specific token values, and have not confirmed whether the cookie content contains HttpOnly/Secure/session values.

6. I continue to look at `bruhealth/sources/egnc/moh/bruhealth/nativeLib/utils/SchemeOpenManager.java`: `handleSchemeRequest` enqueues the Intent. If `AccountManager.getInstance().isLogin()` and top activity is not empty, call `handleDeepLink()`, otherwise start `SplashActivity`. `handleDeepLink()` reads `getSchemeIntent().getData()`. If the host/path contains `jump` and the query string contains `url=`, the string after `url=` will be taken; if `DeepLinkFactory.createDeepLink(strSubstring2)` returns to manager, it will be processed, otherwise `Router.getInstance().handle(strSubstring2, null)`. `handleDeepLink` of `bruhealth/sources/egnc/moh/base/utils/deeplink/NativeDeepLinkManager.java` injects query parameters into ARouter `Postcard.withString(key,value)` one by one. `bruhealth/sources/egnc/moh/base/utils/deeplink/WebDeepLinkManager.java` parses the URL options and then opens `RouteConstants.PAGE_BASE_WEB` and passes in `Constants.WEB_LOAD_URL`. In `bruhealth/sources/egnc/moh/base/utils/deeplink/DeepLinkFactory.java`, null is returned at the beginning of http, and `NativeDeepLinkManager` is returned at the beginning of native. `bruhealth/sources/egnc/moh/bruhealth/scheme/SchemeInit.java` registers `evydbh://open/external`, `evydbh://cordova`, `http://`, `https://`, health, permission, auth, photo, multipicker, IM, TRTC and other handlers. If `system=yes` is set to http/https route in `bruhealth/sources/egnc/moh/bruhealth/scheme/UrlScheme.java`, the system browser will be opened, otherwise `EvydEnvironment.openSelfWebView(...)` will be used.

I checked Chain 8 WebView origin/geolocation/cookies/bridge: WebView settings `CommonWebView.java` `setJavaScriptEnabled(true)`, `setDomStorageEnabled(true)`, `setDatabaseEnabled(true)`, `setGeolocationEnabled(true)`, `setMixedContentMode(0)`, `setAllowFileAccess(true)`, `setAllowContentAccess(true)`, `CookieManager.getInstance().setAcceptThirdPartyCookies(this.mWebView, true)`; `callback.invoke(origin, true, false)` in `ChromeClient.onGeolocationPermissionsShowPrompt(origin, callback)` of Geolocation auto grant; Web request logs record URL, Method, Headers in `CommonWebView.shouldInterceptRequest(...)`; Bridge token in `CallMethodManager` `GET_APP_TOKEN`; Deep link possible entry in `SchemeOpenManager` can hand over the deep link URL to Router/WebView. `setMixedContentMode(0)` is the mode in Android corresponding to 0 that allows mixed content.
