7. The second independent issue I found is I-02: Redux persistent state uses a recoverable static encryption key within the APK, which has a confirmed static design issue. The Redux Persist configuration in the React Native bundle persists `session`, `monitor`, `resources`, `shared` and other states, using `redux-persist-transform-encrypt`. The encryption key comes from the `getKey()` function in the bundle. An attacker can restore the algorithm by getting the APK. I don't have a runtime storage dump, so I can't confirm that the real token file already exists on the current device.

8. I found that the `redux-persist-transform-encrypt` encryption and decryption module exists in the bundle, located at `bupa/resources/com.bupa.digitalprimarycare.apk/assets/index.android.bundle:734`; the persistence configuration whitelist includes `session`, `monitor`, `resources`, `shared`, and uses `{ secretKey: getKey(), onError: ... }` Create a transform and search for `secretKey:`, `getKey`, `whitelist` in the bundle to locate it; there is a transform for `session`, including state overrides such as `isLocked:true`; `monitor` filter fields include `unitPreferences`, `appleHealthSourceConnected`, `appleHealthSyncAnchorMap`, `dashboardMetricFilter`, `metricConversations`, `monitorDisclaimerAccepted`; `resources` filter `userContext`; `getKey()` itself is constructed through obfuscated character operation within the bundle and is still a recoverable static secret within the APK.

10. I think this issue is sensitive because `authToken`, `idToken`, `refreshToken` belong to authentication credentials or identity claim materials, and static built-in keys are not device binding secrets; if an attacker can obtain the App private storage (such as root, debug backup, supply chain image, forensic image), the key in the APK is enough to assist offline decryption; this is not the conclusion of "token words in the string", but the reducer field, persist whitelist and encrypt transform The code-level chain formed by the three. The static data flow can be summarized as follows: login/refresh process generates access token, id token, refresh token → action/reducer writes session state → Redux Persist whitelists session for persistence → encrypt transform encryption → secret key is provided by static `getKey()` in bundle → If the attacker obtains APK and App private storage, encryption is no longer a device binding protection. The boundary is that it is not confirmed that the real token exists in the `AsyncStorage` file in the real device, it is not confirmed whether the token is still valid, and it is not confirmed whether it is completely replaced by Keychain.
