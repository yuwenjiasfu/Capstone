A Empirical Study of 15 Android BLE Health Applications: Comprehensive Findings Report


Introduction

Bluetooth Low Energy (BLE) health and fitness applications form a fast-growing category on Android. A typical app in this class connects to a bonded peripheral (a blood-pressure cuff, a glucometer, a body-fat scale, a fetal-heart monitor, a continuous glucose monitor, or a wearable), reads a stream of physiological measurements over GATT, persists those measurements together with identity and device metadata on the phone, and synchronises them to a vendor cloud or to a third-party health platform. The same app is also the place where the user types in profile data such as name, date of birth, height, and weight, where the access token to the vendor backend is stored, and where third-party analytics, push, and crash-reporting SDKs are initialised.

A single BLE health app therefore exposes three interacting attack surfaces: the BLE link itself, the Android application layer, and the network / cloud layer. Most published audits look at one of these surfaces in isolation. This study takes the opposite view. It applies a single rule-driven static workflow to fifteen real Android BLE health applications and examines, app by app and surface by surface, what conclusions are actually supportable from the shipped code.

The results are reported as category-level empirical roll-ups rather than isolated case studies. For each rule category, we list the affected applications, assign an evidence tier, and expand representative cases with source-level evidence

1. Background

1.1 Why BLE health apps are a distinct target class

Compared with general-purpose Android apps, BLE health applications combine three properties that change their security posture. First, the data density per user is high: a single account routinely carries continuous physiological measurements (blood pressure, blood glucose, oxygen saturation, heart rate, body fat, fetal-heart waveforms, training sessions) plus stable identifiers (account id, device serial, MAC address) plus profile data (name, phone, age, height, weight). Second, the engineering posture around transport encryption is, on average, weaker than in financial or communication apps; many of the apps reviewed here still ship with cleartext traffic permitted at the package level, with mixed HTTP and HTTPS base URLs in shipped assets, or with TLS-bypass code paths on real product flows. Third, the consequence of a leak is more severe, because the leaked record is medical and is paired with full identity. These three properties also make the issues easier to surface from a static review.

1.2 The three layers in scope

The review treats every app under three layers. The BLE link covers GATT scan, connect, characteristic read/write/notify, command framing, and pairing/bonding posture. The Android application layer covers Manifest declarations and exported components, local persistence (SharedPreferences, Room/ObjectBox/SQLite, custom files), crypto, logging, WebView and JavaScript-bridge configuration, third-party SDK initialisation, and the dynamic flags visible in release builds (debug toggles, log strings, build configuration). The network / cloud layer covers HTTP and HTTPS endpoints, the TLS configuration of the clients used to reach them, the authorisation material carried in headers and bodies, and the destinations of third-party telemetry.

1.3 Evaluation criteria (two tiers)

Every finding in this report carries one of two evidence labels. A finding labelled confirmed is observable directly in the shipped artifacts (Manifest, Smali / Java sources, decompiled assets, embedded XML / JSON), and the underlying evidence file lists the exact source file and line number. A finding labelled supported_hypothesis is one whose static chain is closed enough that the risk is consistent and plausible, but whose final empirical step (a captured packet, a real device exchange, an actual exfiltrated byte) was not produced in this study.


2. Methodology

2.1 Application selection

Fifteen Android applications were selected. Two are general BLE explorers and SDK demos: nRF Toolbox and LightBlue. Twelve are vendor-branded health or fitness apps: BLE Health Plus, bruHealth, Bupa Anywhere, FitPro, Yuwell HealthManager, MyVitale, Nutrack, Polar Beat, Rossmax Connect, WeightTrack, Senssun SmartLink, and Yuwell uHealth. One is an open-source diabetes monitor: xDrip+. Each app was required to contain a real BLE pairing or measurement code path in the shipped APK, to be available for download, and to be in current distribution at the time of the review.

2.2 Two literature databases

Two distinct sets of papers were used and they play different roles in the workflow. The first, called database_original, was assembled in the early phase of the project before the workflow was settled. It is a relatively unfiltered collection of papers covering possible privacy and security problems in mobile health and IoT applications. Its purpose was to build personal background knowledge of the threat landscape. The free-form notes in finding.txt files were written from this material together with my own intuition and prior knowledge, and they are therefore the more subjective layer of the project record.

The second, called database_main, was assembled later once the workflow was being formalised. Each paper in database_main was selected because its findings are concrete enough to be expressed as a discrete rule. Each rule was then manually translated into a small set of search keywords (Java identifiers, Manifest attributes, characteristic UUIDs, crypto suite names, file-system patterns) consumed by Python scripts that scan the decompiled APK with ripgrep. Every machine hit was then verified by hand against the source line. The summary_notes.txt file produced for each app is the output of that path: every issue listed in summary_notes.txt maps back to a specific rule and to a specific paper in database_main. This report draws primarily from that material.

2.3 Per-application pipeline

Each APK was unpacked and processed under the same pipeline. The package was decoded with apktool to recover the Manifest and resources, and decompiled with JADX to recover Java sources; assets such as embedded HTML, JavaScript, JSON configuration, and SQL migrations were extracted directly. The rule keywords were then run as Python ripgrep searches across the decoded tree, and every hit was opened, read in context, and either confirmed against the rule or rejected. A finding was categorised under the rule whose keyword produced the hit, and assigned an evidence tier of confirmed or supported_hypothesis according to the criteria in section 1.3.


3. Results

The results are organised as category roll-ups. Each subsection opens with a short pattern description, lists all apps in which the issue appears with an evidence tier, and then expands a small number of representative cases with file-and-line evidence.


3.1 BLE link layer

Across the vendor-branded apps the BLE code path is structurally similar: scan for advertisements that match a vendor service UUID, connect with GATT, enable a notify characteristic, decode the incoming bytes into a measurement object (blood pressure, glucose, SpO2, weight, heart rate, fetal-heart audio frame, training session), and forward that object to local storage and to the upper layer. The reverse direction — commands such as history sync, time set, Wi-Fi configuration, Wi-Fi list query, OTA, device reset, lock / unlock, payment-QR push — is also present in several apps.

Confirmed: the GATT characteristic and command surface is exposed and reachable from a first-party caller in every vendor app: BLE Health Plus (BaseBleActor + BleConnectionService), bruHealth (GlucometerBridge), FitPro (BaseLeService + BaseReceiveData + SendData), Yuwell HealthManager (BPMManager, GlucoseManager, OximeterManager, ThermometerManager, FetusManager), MyVitale (OmronSessionManager + SessionController), Nutrack (classic-Bluetooth RFCOMM SPP and Wi-Fi TCP sockets — not GATT — with Balanza control commands ":L", ":M", ":N"), Polar Beat (PFTP protocol on fixed server UUIDs and MTU, BR063/066 confirmed), Rossmax (VUBleManager), WeightTrack (no first-party BLE path, excluded from this category), xDrip+ (DexCollectionService + transmitter characteristics), Senssun SmartLink (GATT scan/connect/notify with command helper), Yuwell uHealth (MoxManager + BLE callbacks), nRF Toolbox and LightBlue (general explorer surfaces).

Supported_hypothesis: application-layer authentication of the peer (HMAC, signature, challenge-response, per-session keying, device certificate check, replay defence) is not visible in the first-party code of any of these apps. The attack-style rules (BR061–BR075, rule49–55 / 61 / 63, R073–R084) therefore remain supported, not confirmed, because the static review cannot prove that the real peripheral accepts spoofed, replayed, or injected frames.

Representative detail. In Yuwell uHealth, MoxManager and UHBleCallbacksHandler parse BP, glucose, and SpO2 measurements directly into measurement models with no application-layer HMAC or signature step before the bytes move inward (BR066 confirmed; BR061-BR065, BR067-BR069, BR073-BR075 supported_hypothesis). In FitPro, SendData.java:314 builds the generic protocol frame, SendData.java:294 builds pairing, :123-125 sets user information, :405-406 builds payment payloads, and BaseReceiveData.java:1798-1799 calls uploadHeartBlood(measureDetailsModel) immediately after saving the measurement — none of which is wrapped with a visible HMAC / signature / nonce layer. In Polar Beat, BluetoothService + clq.java:18-23 + clr.java:44-48 route writes by Bluetooth device address and characteristic UUID into a PFTPController; the PFTP protocol surface is confirmed but the attack-style rules (BR061/062/064/067/068/069/071/074/075) stay supported. In Senssun SmartLink, a command helper builds Wi-Fi configuration, Wi-Fi list, Wi-Fi OTA, and history-sync commands without an obvious application-layer signature gate, so the "visible app path does not show an obvious app-layer MAC, signature, or provenance proof" conclusion is explicitly retained in rule49 / 50 / 51 / 53 / 54 / 55 / 61 / 63.

Nutrack is a special case. Its device link is classic Bluetooth RFCOMM plus Wi-Fi TCP, not GATT: Bluetooth.java:7-8 fixes the SPP UUID and port 5373, BluetoothClient.java:54 and BluetoothServer.java:45 use insecure RFCOMM, WiFiClient.java:35,54 and WiFiServer.java:28,61 use ordinary sockets, ProtoFeedMsg.java:92-115,139-143 uses only unkeyed MD5 integrity, ProtoFeed.java:314-350,520-576 sends STX + JSON + ETX frames. The attack surface is adjacent to the BLE category rather than inside it, but the application-layer gap — no HMAC, no signature, only keyless MD5 — is the same.


3.2 Application reverse-code analysis (static and dynamic)

3.2.1 Cleartext transport policy and TLS posture

Package-level cleartext permission is one of the most consistent confirmed findings of the set.

Confirmed cleartext-permitted policy: FitPro (AndroidManifest.xml:154 usesCleartextTraffic="true" plus network_security_config.xml:3 and :13 both allowing cleartext, BR006/BR013 confirmed), Nutrack (AndroidManifest.xml:61 usesCleartextTraffic="true", BR006 confirmed), Bupa Anywhere (AndroidManifest.xml:125 usesCleartextTraffic="true", BR006 confirmed), xDrip+ (AndroidManifest.xml:139 usesCleartextTraffic="true", BR006/BR012 confirmed), Yuwell uHealth (network_security_config.xml:3 cleartextTrafficPermitted="true", BR006/BR008 confirmed), Senssun SmartLink (usesCleartextTraffic="true" plus HTTP business URLs shipped in the bundle, rule06 confirmed), Yuwell HealthManager (global cleartext allowance kept as broader posture, supported).

Confirmed code-level TLS bypass on real product paths:

xDrip+ — ShareRest.java:98 has an empty checkServerTrusted; ShareRest.java:140-143 returns true unconditionally from the HostnameVerifier; the same client is used at ShareRest.java:174-176 for Dexcom Share authentication (account name and password) and at ShareRest.java:204-208 to upload BG records (BR009 / BR010 confirmed).

Polar Beat — bqh.java:33 builds the DefaultHttpClient; bqh.java:38 sets ALLOW_ALL_HOSTNAME_VERIFIER; bqi.java:16-17 installs a custom TrustManager; bqj.java:16,20 leaves checkClientTrusted empty and has only weak server-certificate logic; bqf.java:21,25,26 uses that client to send /register/user with username and password (BR009 / BR010 / BR012 confirmed).

bruHealth — the StepsUploader path uses an SSLSocketClient with an empty checkServerTrusted plus a HostnameVerifier that returns true, installed through OkHttpUtils.setSafe; the StepsUploader uses that client to upload step and calorie data together with wlapp-token and device-id. The main ApiClient path is weaker but also real: EvydHostnameVerifier returns true for the configured base host branch (BR009 / BR010 / BR012 confirmed).

Apps without completed evidence for this category: nRF Toolbox, LightBlue, MyVitale, Rossmax, BLE Health Plus, WeightTrack, Yuwell HealthManager, Yuwell uHealth, Senssun SmartLink, FitPro, Nutrack, Bupa Anywhere do not show a first-party trust-all TrustManager or an always-true HostnameVerifier on a business client; those TLS-bypass rules stay out for these apps.

3.2.2 Hardcoded crypto keys and weak modes

Confirmed hardcoded-key / weak-mode cryptography on real product paths:

Yuwell uHealth — EncryptUtil.java:22 hardcodes "8yu&well"; line 56 decrypts with DES/ECB/NoPadding; line 85 encrypts with DES/ECB/PKCS7Padding; SyncService.java:120,131 wraps the upload payload with that encrypt call and DatabaseServiceImpl.java:1498,1508 decrypts incoming server data with the same utility, so this is part of the live sync path and not dead code (BR017 confirmed).

xDrip+ — Backup.java:621 uses AES/CBC/PKCS5Padding with the key 5861bdb93f5ba8effa43c8ff7f26c3be hardcoded at Backup.java:623 (BR017 confirmed).

BLE Health Plus — AESCipher uses IV_STRING = "BlupixelHealthPl" and Cipher.getInstance("AES") (mode/padding not specified, implying a weak default); the same cipher wraps the full Room database that BackUpFragment checkpoints, writes as ble_health_backup.bkup under externalCacheDir, and uploads to Google Drive (R051 / R054 / R057 / R059 / R060 supported through this backup chain).

Nutrack — Encrypter.java:15-16 defines a fixed IV and a fixed AES key; lines 19-23 configure AES/CBC/PKCS5PADDING; MainActivity.java:1044-1049 reads NutrackLicense, decrypts it, and extracts lic_uuid (BR017 confirmed).

The boundary is identical in all four cases: the existence of weak crypto inside the client does not prove that the outer transport is missing or that the server has no other check, but it does confirm a fixed-key DES or fixed-IV AES design inside the client sync / backup chain.

3.4 Multi-layer combined issues

This section keeps only chains that are not visible from any single layer in isolation. The chains below are the ones that earned their place in the report because each component on its own would not have been alarming under any single rule in database_main, and the risk only emerges when two or three components are joined.

(a) Yuwell HealthManager — fetal BLE → toString() → release log. A FetalHeartMeasurement object is built from raw fetal-heart audio frames received over BLE. The release-build code path emits this object into the standard Android log pipeline through a default toString() rendering (FetalHeartMeasurement.java:19,26-30 include value, measure time, and local file paths in toString()), on a logger that the same app keeps active in release (FetusHome.java:60). Looking only at the BLE callback would not raise an issue (the parser is correct); looking only at the logger would not raise one either (it is a generic Log.i). The risk only appears once the two are joined: an unencrypted, vendor-private fetal-heart waveform leaves the device-bound BLE channel and lands on a writable log surface inside the app.

(b) Senssun SmartLink — external app.zip → UNI Web layer replacement. The Manifest exposes a deep-link entry on the smartlink:// scheme. The embedded UNI WebView layer that drives the business UI is loaded from a .zip archive on the device, and access to the location of that .zip is not gated by an exclusive permission. Either component on its own is a small finding: deep-link entries are common, and zip-driven web layers are common. Together, they form a supply-chain entry into a health app: a process that drops a same-named .zip into the loader path can swap the entire web business layer the next time the user opens the app.

(c) Senssun SmartLink — flutter.sp_token across local file, log, and remote JS. The same authentication token reaches three different tiers in the same app. It is persisted in flutter.sp_token; it is then printed at native MainActivity.initData() through Log.i (rule41 confirmed); and the member-points/web.js asset embeds it into a remote sharecn.senssun.com URL builder. Each step is a small finding in its own rule, but the chained exposure of one long-lived credential at a local file, in logcat, and inside a remotely loaded JS surface, is materially worse than any of the three taken alone.

(d) Yuwell uHealth — DES sync + cleartext-permitted base + log of decrypted upload. The hardcoded-DES sync wrapper (BR017 confirmed) is not by itself a credential leak. The cleartext-permitted base policy (BR006 confirmed) is not by itself proof of misuse. The log line that prints the decrypted upload JSON together with accessToken and deviceId (BR047 confirmed) is what closes the chain: anyone with logcat-level access on the device can read what the server would have received, including the authentication material, regardless of the DES wrapper or the outer transport.

(e) BLE Health Plus — fixed-IV AES + encrypted DB backup + Google Drive destination. The fixed-IV cipher is the same cipher used to wrap the local SQLite backup that the app uploads to the user's Google Drive. The fixed IV alone is a code-quality finding; the third-party backup destination alone is a feature. The combination means the encrypted backup objects sitting in users' Drive accounts can be decrypted with key material that is shipped inside the public APK.

(f) Nutrack — cleartext sync + classic Bluetooth / Wi-Fi sockets + Crashlytics custom keys. Nutrack's Crashlytics setCustomKey calls embed account-side identifiers (usuario, android_id, cuenta, licencia). The same app sends sync over a cleartext channel, uses classic Bluetooth and Wi-Fi sockets with unkeyed MD5 integrity, and also sends android_id as dev_id in the SYNC.php query string. Neither the Crashlytics call, nor the cleartext sync, nor the classic Bluetooth path is by itself a high-severity privacy problem; together they create a cross-tier path where an account identifier reaches a third-party crash backend at the same time as the same identifier travels in the clear over the air.


4. Conclusion

Across the fifteen Android BLE health applications reviewed, the most consistent confirmed findings cluster into six categories: cleartext transport at the package-policy level (FitPro, Nutrack, Bupa, xDrip+, Yuwell uHealth, Senssun SmartLink, with Yuwell HealthManager as broader posture); hardcoded crypto keys on real product paths (Yuwell uHealth DES, xDrip+ AES backup, BLE Health Plus fixed-IV AES, Nutrack fixed AES); plain local persistence of credentials and health data (all vendor apps in varying degrees, with nRF Toolbox and LightBlue still holding at least custom macro storage and Adafruit secret material); release-build logging that mixes sensitive data with identifiers (MyVitale health values, Yuwell HealthManager fetal paths, Yuwell uHealth sync + BLE, xDrip+ Dexcom credentials, Senssun SmartLink token, FitPro token + BLE + health, WeightTrack OAuth, bruHealth sinks); exported components and WebView / JS bridges with controllable inputs (Yuwell uHealth WebView, FitPro debug / command / upload services, Bupa video consultation, BLE Health Plus share report, Yuwell HealthManager statistics WebViews, Rossmax onJsPrompt); and a dense third-party SDK identifier surface (Bupa AppCenter / Braze / Branch / Sentry, FitPro QQ / Sina / Tencent / Umeng / WeChat, LightBlue Mailchimp + Firebase, Senssun SmartLink JPush + UMeng, Yuwell uHealth JPush / UMeng / Udesk / Youzan, nRF Firebase with gating, Polar Beat FB + GA ADID).

The most consistent supported_hypothesis findings cluster differently. They concern BLE command injection, replay, and spoofing in apps where no application-layer signature gate is visible between the BLE bytes and the local model (all vendor apps plus Nutrack's classic-Bluetooth analogue); pre-consent or undisclosed third-party SDK telemetry in apps where the static SDK initialisation is wired but the actual egress bytes were not captured; and the eventual cloud delivery of credentials and health values whose static client-to-server chain is closed but whose wire bytes were not captured. The two-tier evidence framing is what keeps these distinct from the confirmed set.

The multi-layer chains gathered in section 3.4 illustrate the most important point of the study. Each chain is held together by elements that, taken in isolation, would not have triggered any single rule strongly. A fixed IV is a code-quality issue; a Google Drive destination is a feature; a Log.i call is a debug habit; a deep-link entry is a Manifest convenience. It is the path between them that creates the actual exposure. A review that scans only one layer at a time is structurally unable to see those paths, which is why a uniform cross-layer workflow is the most useful contribution this review can make to the next iteration of the analysis.
